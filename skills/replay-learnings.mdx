## Overview

Replay Learnings surfaces relevant past lessons before you start working on a task. It searches your correction history, recalls past mistakes, and applies prior patterns — like muscle memory for your coding sessions.

<Note>
**Think before you act.** Surface what you've learned before making the same mistake twice.
</Note>

## Trigger

Use when:
- Starting a new task
- Saying "what do I know about", "before I start", "remind me about"
- After switching contexts or taking a break
- Working on an area with high past correction rate

<CodeGroup>
```bash Claude Code
/replay what do I know about auth
/replay before I start the API refactor
/pro-workflow:replay authentication patterns
```

```bash Cursor
# Skill is available automatically
"what have I learned about testing?"
"replay my learnings about database migrations"
```
</CodeGroup>

## Workflow

<Steps>
  <Step title="Extract Keywords">
    Extract keywords from the task description
    
    Example:
    - "auth refactor" → `auth`, `middleware`, `refactor`
    - "add webhook retry" → `webhook`, `retry`, `async`
  </Step>
  
  <Step title="Search Learnings">
    Search memory for matching patterns
    
    ```bash
    grep -i "auth\|middleware" .claude/LEARNED.md 2>/dev/null
    grep -i "auth\|middleware" .claude/learning-log.md 2>/dev/null
    grep -A2 "\[LEARN\]" CLAUDE.md | grep -i "auth\|middleware"
    ```
  </Step>
  
  <Step title="Check Session History">
    Look for similar work in past sessions
    
    - What was the correction rate?
    - How many edits vs corrections?
    - Any recurring patterns?
  </Step>
  
  <Step title="Rank by Relevance">
    Surface the top learnings ranked by:
    - Keyword match strength
    - Application frequency
    - Recency (newer = more relevant)
    - Correction impact (high correction rate = important)
  </Step>
  
  <Step title="Provide Briefing">
    Output a structured briefing with:
    - Top 3-5 relevant learnings
    - Session history for similar work
    - Suggested approach based on patterns
  </Step>
</Steps>

<Info>
If no learnings match, say so explicitly rather than forcing irrelevant results. Suggest using the scout agent to explore first.
</Info>

## Output Format

```text
REPLAY BRIEFING: <task>
=======================

Past learnings (ranked by relevance):
  1. [Testing] Always mock external APIs in auth tests (applied 8x)
     Mistake: Called live API in tests, caused flaky failures
     
  2. [Navigation] Auth middleware is in src/middleware/ not src/auth/ (applied 5x)
  
  3. [Quality] Add error boundary around auth state changes (applied 3x)

Session history for similar work:
  - 2026-02-01: auth refactor — 23 edits, 2 corrections (8.7% rate) ✓
  - 2026-01-28: auth middleware — 15 edits, 4 corrections (26.7% rate) ⚠️
    ^ Higher correction rate — review patterns before starting

Suggested approach:
  - Mock external APIs (learning #1)
  - Check src/middleware/ first for auth code (learning #2)
  - Plan error handling strategy upfront (learning #3)
```

  ⚠️ High correction rate on Jan 28. Review what went wrong:
     - Forgot to mock auth provider (learning #1)
     - Looked in wrong directory first (learning #2)
     - Put logic in components instead of middleware (learning #3)

Suggested approach:
  1. Start in src/middleware/auth.ts (learning #2)
  2. Set up test mocks first (learning #1)
  3. Keep session logic in middleware (learning #3)
  4. Review Jan 28 session notes for additional context

Ready to proceed with this context?

✓ Recent migrations went well. Pattern is working.

Suggested approach:
  1. Write both up and down migrations (learning #2)
  2. Test rollback in local environment (learning #3)
  3. Run on staging first (learning #1)
  4. Don't modify existing migrations (learning #4)
  5. Follow the pattern from Feb 28 session (zero corrections)

Confident to proceed with migrations?

Session history:
  - 2026-03-01: Optimize table rendering — 18 edits, 5 corrections (27.8% rate) ⚠️
  - 2026-02-20: Fix slow search — 12 edits, 3 corrections (25% rate) ⚠️
  - 2026-02-05: Dashboard performance — 20 edits, 6 corrections (30% rate) ⚠️

⚠️ HIGH CORRECTION AREA (avg 27.6% correction rate)

Common mistakes in past sessions:
  - Optimizing without profiling first (learning #3)
  - Testing with small datasets (learning #5)
  - Over-using memoization (learning #3)

Suggested approach:
  1. ⚠️ PROFILE FIRST with React DevTools (learning #3)
  2. Test with realistic data (learning #5)
  3. Start with the highest impact optimizations
  4. Apply memo/useMemo only where profiler shows benefit
  5. Capture new learnings — this area needs more patterns

Proceed with extra caution in this area?

## Configuration

### Auto-Replay on Session Start

Add to CLAUDE.md:

```markdown
## Session Start

Before starting any task >5 files:
1. Run replay-learnings with task keywords
2. Review top 3 learnings
3. Acknowledge high correction areas
4. Proceed with context
```

### Custom Relevance Scoring

In settings (Claude Code):

```json
{
  "learnings": {
    "replay": {
      "rankBy": ["relevance", "frequency", "recency"],
      "maxResults": 5,
      "includeMistakes": true
    }
  }
}
```

## Troubleshooting

### No Learnings Returned

Check:
1. Are learnings stored in correct location (LEARNED.md or database)?
2. Try broader keywords: "auth" instead of "OAuth middleware"
3. Check if learnings exist: `/list`

### Irrelevant Results

Refine search:
```bash
# Too broad
/replay about testing

# More specific
/replay about testing auth endpoints
```

### Stale Learnings Surfaced

Remove outdated learnings:
```bash
# View all learnings
/list

# Remove stale ones from LEARNED.md or database
# Or mark as archived
```
