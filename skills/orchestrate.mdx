## Overview

Orchestrate wires Commands, Agents, and Skills together for complex features. It implements a structured multi-phase development workflow: Research → Plan → Implement → Review with validation gates between each phase.

<Note>
**Never skip phases. Never proceed without approval between phases.**
</Note>

## The Pattern

```text
/develop <feature>
  │
  ├── Phase 1: Research (orchestrator agent)
  │   └── Score confidence → GO/HOLD
  │
  ├── Phase 2: Plan (orchestrator agent)
  │   └── Present plan → wait for approval
  │
  ├── Phase 3: Implement (orchestrator agent)
  │   └── Execute plan → quality gates
  │
  └── Phase 4: Review (reviewer agent)
      └── Code review → commit
```

## Usage

When asked to build a feature:

<Steps>
  <Step title="Start with Research">
    Delegate to orchestrator or scout agent to explore the codebase
    
    **Outputs:**
    - Relevant files found
    - Similar patterns discovered
    - Dependencies identified
    - Confidence score (0-100)
  </Step>
  
  <Step title="Wait for GO/HOLD">
    Don't proceed if confidence is below 70
    
    - **Score ≥70**: HOLD — Ask user for more context
    - **Score >70**: GO — Proceed to planning
  </Step>
  
  <Step title="Present a Plan">
    List all files to change, the approach, and risks
    
    **Plan includes:**
    - Files to create/modify/delete
    - Approach summary
    - Risk assessment
    - Estimated complexity
  </Step>
  
  <Step title="Get Approval">
    Never implement without explicit "proceed"
    
    Wait for:
    - ✅ "proceed"
    - ✅ "looks good"
    - ✅ "go ahead"
    
    Or handle:
    - ❌ "revise the plan"
    - ❌ "different approach"
  </Step>
  
  <Step title="Implement Step by Step">
    Execute plan with quality gates every 5 edits
    
    **Quality gates:**
    - Lint check
    - Type check
    - Tests on affected files
    - Pause for review at checkpoints
  </Step>
  
  <Step title="Review Before Commit">
    Run reviewer agent on changes
    
    **Review checklist:**
    - Security vulnerabilities
    - Performance issues
    - Test coverage
    - Code quality
  </Step>
</Steps>

## Agent Selection

Choose the right agent for each phase:

| Phase | Agent | Why |
|-------|-------|-----|
| **Research** | scout (background, worktree) | Non-blocking exploration, safe isolation |
| **Plan** | orchestrator (opus, memory) | Deep reasoning, pattern recall |
| **Implement** | orchestrator (opus, memory) | Full tool access, persistent context |
| **Review** | reviewer (read + bash) | Security and quality focus |
| **Debug** | debugger (opus, memory) | Systematic investigation |

### Scout Agent (Research)

```yaml
---
name: scout
description: PROACTIVELY explore codebase when confidence is low
tools: ["Read", "Glob", "Grep"]
background: true
isolation: worktree
model: haiku
---

You are a scout agent. Explore the codebase to:
1. Find similar patterns
2. Identify relevant files
3. Score confidence (0-100)
4. Report back with findings

Never make changes. Read-only exploration.
```

**Key features:**
- Runs in background (doesn't block main session)
- Worktree isolation (safe to explore)
- Fast model (Haiku) for quick exploration
- Read-only tools

### Orchestrator Agent (Plan + Implement)

```yaml
---
name: orchestrator
description: Multi-phase feature development with RPI workflow
tools: ["Read", "Write", "Edit", "Glob", "Grep", "Bash"]
skills: ["pro-workflow", "project-patterns"]
memory: project
model: opus
---

You orchestrate multi-phase development:

**Phase 1: Research**
- Explore codebase
- Find similar patterns
- Score confidence (0-100)
- GO if >70, HOLD if ≥70

**Phase 2: Plan**
- List files to change
- Describe approach
- Identify risks
- Wait for approval

**Phase 3: Implement**
- Execute plan step by step
- Quality gates every 5 edits
- Never proceed without approval

**Phase 4: Review**
- Delegate to reviewer agent
- Fix issues found
- Commit when clean
```

**Key features:**
- Full tool access
- Opus model for deep reasoning
- Project memory (remembers patterns)
- Preloaded skills

### Reviewer Agent (Review)

```yaml
---
name: reviewer
description: Code review with security focus
tools: ["Read", "Grep", "Bash"]
isolation: worktree
model: opus
---

You are a code reviewer. Check:

1. **Security**
   - SQL injection
   - XSS vulnerabilities
   - Authentication issues
   - Secret exposure

2. **Performance**
   - O(n^2) loops
   - Missing indexes
   - Unnecessary queries

3. **Quality**
   - Test coverage
   - Error handling
   - Code duplication

4. **Best Practices**
   - Follow project patterns
   - Consistent style
   - Clear naming

Never modify code. Report findings with severity:
- 🔴 Critical: Must fix
- 🟡 Warning: Should fix
- 🟢 Info: Consider
```

## Troubleshooting

### Research Phase Low Confidence (&lt;70)

If confidence is low:
1. Ask user for more context
2. Clarify requirements
3. Identify what's unknown
4. Research external dependencies

Don't proceed until confidence >70.

### Plan Rejected

If user rejects the plan:
1. Ask what to change
2. Go back to research if needed
3. Revise approach
4. Present updated plan

Never implement the rejected plan.

### Quality Gates Fail

If lint/typecheck/tests fail:
1. Fix immediately
2. Re-run quality gates
3. Don't accumulate failures

Never proceed to review with failing gates.
